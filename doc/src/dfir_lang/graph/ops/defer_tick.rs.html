6/// Buffers all input items and releases them at the next time boundary.
7///
8/// When used at the top level or inside a root-level `loop { }` block (which is fused with
9/// the tick), data is deferred to the next tick. When used inside a nested `loop { }` block,
10/// data is deferred to the next loop iteration.
11///
12/// The presence of buffered data will cause the next tick or loop iteration to fire (non-lazy).
13///
14/// See the [book discussion on time](../concepts/life_and_times) for details on ticks.
15///
16/// `defer_tick` is sometimes needed to separate conflicting data across time,
17/// in order to preserve invariants. Consider the following example, which implements
18/// a flip-flop -- the invariant is that it emit one of true or false in a given tick
19/// (but never both!)
20///
21/// ```rustbook
22/// pub fn main() {
23///     let mut df = dfir_rs::dfir_syntax! {
24///         source_iter(vec!(true))
25///                 -> state;
26///         state = union()
27///                 -> assert(|x| if context.current_tick().0 % 2 == 0 { *x == true } else { *x == false })
28///                 -> map(|x| !x)
29///                 -> defer_tick()
30///                 -> state;
31///     };
32///     for i in 1..100 {
33///         println!("tick {}", i);
34///         df.run_tick();
35///     }
36/// }
37/// ```
38///
39/// `defer_tick` can also be handy for comparing stream content across ticks.
40/// In the example below `defer_tick()` is used alongside `difference()` to
41/// filter out any items that arrive from `inp` in the current tick which match
42/// an item from `inp` in the previous
43/// tick.
44/// ```rustbook
45/// // Outputs 1 2 3 4 5 6 (on separate lines).
46/// let (input_send, input_recv) = dfir_rs::util::unbounded_channel::<usize>();
47/// let mut flow = dfir_rs::dfir_syntax! {
48///     inp = source_stream(input_recv) -> tee();
49///     inp -> [pos]diff;
50///     inp -> defer_tick() -> [neg]diff;
51///     diff = difference() -> for_each(|x| println!("{}", x));
52/// };
53///
54/// for x in [1, 2, 3, 4] {
55///     input_send.send(x).unwrap();
56/// }
57/// flow.run_tick();
58///
59/// for x in [3, 4, 5, 6] {
60///     input_send.send(x).unwrap();
61/// }
62/// flow.run_tick();
63/// ```
64///
65/// You can also supply a type parameter `defer_tick::<MyType>()` to specify what items flow
66/// through the the pipeline. This can be useful for helping the compiler infer types.
